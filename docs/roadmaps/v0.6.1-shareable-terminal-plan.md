# v0.6.1 → v1.0.0: ResearchBets shareable terminal execution plan

## Why this plan exists

This plan turns the current hypothesis validation into a concrete execution gate:

- Keep the existing terminal architecture and provenance spine.
- Remove trust-breaking dead-air states from the first fold.
- Unify mode semantics and board truth across `/` and `/today`.
- Promote slip risk from hidden heuristics to an explicit, truthful panel.
- Build toward measurable calibration and outcome feedback for serious bettors.

## North-star success criteria

ResearchBets should feel like a **serious decision terminal**, not a picks feed:

1. First fold always shows useful board proof in demo/cache/degraded conditions.
2. Users can trust mode labels (`demo`, `cache`, `live`) everywhere.
3. Slip analysis explicitly surfaces shared exposure + weakest-leg risk.
4. Outcome feedback and calibration are visible enough to influence behavior.

## Ship gate: v0.6.1 (shareable terminal)

### 1) Canonical today truth resolver (blocker)

Create a server-only resolver shared by `/api/today`, `/`, and `/today` that:

- Applies mode policy once.
- Guarantees non-empty board rows in demo/fallback paths.
- Preserves provenance (`trace_id`, provider health, degrade reason).

**Definition of done**

- `GET /api/today?mode=demo` returns `mode: demo` and `board.length >= 6`.
- Provider failure path returns degraded payload with non-empty board.

### 2) Board parity: one model source for `/` and `/today` (blocker)

Both surfaces must render from the same normalized `board` rows.

**Definition of done**

- Identical row IDs for the same spine context on both pages.
- No page-specific board reconstruction as primary path.

### 3) Mode semantics unification (blocker)

Adopt one presentation helper from mode policy for labels/tooltips.

**Definition of done**

- Cache is never labeled as “Live mode”.
- Run status, spine chips, and trackers use the same helper.

### 4) First-fold noise removal (high)

Use compact `RunStatusPill` on first-fold board surfaces and de-emphasize “quiet/no slate” framing when demo/cache has deterministic rows.

**Definition of done**

- Primary board fold does not lead with “waiting/quiet” messaging in demo/cache.
- Focus is lead rows + state pill.

### 5) Slip Risk Panel v1 (high)

Introduce an explicit panel using existing intelligence outputs:

- Volatility tier
- Shared exposure score (not overstated as statistical correlation)
- Weakest leg hint
- Concentration cues

**Definition of done**

- Panel renders for any slip with ≥1 leg.
- Empty state gives neutral guidance (e.g., add 2–3 legs).
- Copy avoids hype language and guarantees.

## v1.0.0 roadmap focus (post-gate)

### A) Variance taxonomy module

Canonical mapping:

`market type -> variance tier -> typical failure reasons`

Wire this to lead rows, slip legs, and outcome review notes.

### B) Correlation language + estimation maturity

- Keep UI language at “shared exposure/script coupling” until statistically estimated correlation is implemented.
- Add minimally defensible estimation path and confidence caveats before stronger claims.

### C) Calibration as product surface

Expose model calibration with sample-size guardrails:

- Bucketed predicted vs realized
- Brier trend
- Explicit “insufficient sample” state

### D) Behavioral guardrails (non-moralizing)

Add one neutral checkpoint for high-volatility rebuilds after elimination:

- Prompt to remove weakest leg first.
- No punitive lockouts.

## Test and CI additions

Add a fast invariant suite that must pass for shareability:

1. `todayTruth` invariant: non-empty board in demo/fallback.
2. `boardParity` invariant: `/` and `/today` row ID parity.
3. `modeLabels` invariant: cache never shown as live.
4. `firstFold` invariant: no dead-air lead copy in demo/cache.

Keep existing gates:

- `npm run lint`
- `npm run typecheck`
- `npm run test`
- `npm run build`

## Execution order (six sprints)

1. **Sprint 1:** v0.6.1 gatefixes (today resolver, parity, mode semantics, first-fold pill).
2. **Sprint 2:** Slip Risk Panel v1 + weakest-leg action.
3. **Sprint 3:** Variance taxonomy v1 + outcome logging by tier.
4. **Sprint 4:** Shared-exposure estimation v1 + terminology cleanup.
5. **Sprint 5:** Calibration dashboard + sample-size messaging.
6. **Sprint 6:** v1.0 hardening, onboarding narrative, retention/outcome instrumentation.

## Product posture and GTM implication

Near-term wedge should remain **power-user SaaS** (serious bettor workflow) while infra/API positioning matures in parallel.

Reason:

- Repo already has strong end-user loop closure and deterministic UX contracts.
- Statistical claims still need measured proof surfaces before infra-first positioning is credible.
